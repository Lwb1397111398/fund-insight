## 主动跳过的项（附理由）

| 项 | 理由 |
| --- | --- |
| S1 密码硬编码 | **用户明确要求保持原样**，私人使用无敏感数据 |
| D4 删 scheduler 常驻线程 | 本地 CLI `python -m src.tasks.scheduler` 仍用 start/running；生产 Rendr Cron 覆盖了功能。仅修正 health 语义，不删线程逻辑（保守方案，避免破坏本地开发） |
| D7 删 5 张表 MarketData/PolicyData/SentimentData/MarketEvent/UserProfile | 破坏性 schema 变更；advice_service 仍有 UserProfile/MarketData 引用路径需先剥离；空表占 Supabase 额度极小。风险 > 收益，留作未来 |
| SectorFundFlow 唯一约束 | 加 UniqueConstraint 需迁移，生产 `ENABLE_STARTUP_MIGRATIONS=false` 不会自动建；调度不重叠（GitHub 13:30 vs Render 10:30 北京）并发概率低。性价比低，留作未来 |
| P4 `_run_fund_update` 冗余 days=1 请求 | real severity low；"复用 latest"跨方法传递有回归风险；每天 1 次省 1/3 HTTP 收益边际 |
| P5 verify_expired 与 verify_all 冗余重试 | 改隔日排程需引入状态增加复杂度；每天多 1 次空 SELECT 成本可忽略 |

## 5 个被驳回的"发现"（误报，未动）

1. `match_fund_for_prediction` 全表扫描 —— 实有多重短路，基本不触发
2. Post/Prediction `lazy=selectin` + `joinedload` 冲突 —— 实际互斥不叠加
3. Viewpoint 表死列 / `calculate_weight` 无调用 —— 实际被 update_viewpoint_analysis 调用
4. `verify_prediction` 每条调 LLM —— 有 `should_call_llm` 门控
5. `fund_sync_manager` 死代码 —— 被 fund_service 和 `/sync-sector-mapping` 路由调用（静态索引漏了延迟 import）
